# v0.1.1 Triage

This guide is for turning beta and pilot feedback into a small, useful `v0.1.1`.

## Issue classes

| Class | What it means | Example |
|---|---|---|
| Onboarding | The repo is hard to start or understand | “I could not get the quickstart green” |
| Docs clarity | The docs are unclear or misleading | “The allowlist explanation did not match the failure” |
| CI / Action | GitHub Action setup or packaging friction | “The workflow template did not run as pasted” |
| Config / precedence | `docs-drift.config.json` or overrides behave unexpectedly | “Action input did not override config as expected” |
| HTTP runtime | `runner-http` behavior is wrong or missing | “Timeout/redirect/body handling is incorrect” |
| False positives / flaky verification | The tool reports drift that should not fail | “A stable example fails intermittently” |
| Missing capability vs actual bug | The feature is absent, but behavior is expected | “We need another curl flag, but current docs are honest” |

## What to fix before `v0.1.1`

Prioritize in this order:

1. onboarding blockers that stop a third repo from finishing setup
2. CI / Action issues that prevent a clean first integration
3. config or precedence bugs that make the tool feel unpredictable
4. runtime bugs that cause wrong results or confusing failures
5. docs clarity issues that repeatedly slow users down

## What can wait

These are usually not `v0.1.1` blockers unless they block adoption:

- new runtime features that are simply not implemented yet
- broad `curl` compatibility expansions
- non-essential UX polish
- extra examples that do not change adoption success

## Bug vs feature request

Treat an item as a bug if:

- the documented behavior does not match reality
- the failure prevents a pilot repo from getting to a useful state
- the tool behaves inconsistently across local and CI without explanation

Treat it as a feature request if:

- the behavior is honest and documented
- the pilot can still succeed without it
- the request expands scope rather than fixing adoption friction

## Practical triage questions

When a report arrives, answer these first:

1. Did the user start from `README.md`, `docs/user-guide/getting-started.md`, or the pilot guide?
2. Was the issue in local, CI, or GitHub Action usage?
3. Which config file or Action inputs were set?
4. Which snippet failed?
5. Is the problem docs, config, runtime, or workflow packaging?
6. Did the user find a workaround?

## How to decide priority

Fix immediately if:

- a second repo would likely hit the same blocker
- the issue prevents the pilot from completing
- the fix is small and clearly improves adoption

Queue for later if:

- the issue is local to one repo's unusual setup
- the pilot can proceed with a small workaround
- the item is actually a request for broader future capability

## Output format for triage

For each issue, keep a short note with:

- class
- severity
- root cause
- workaround
- next action

That is enough to decide whether `v0.1.1` should fix it.
